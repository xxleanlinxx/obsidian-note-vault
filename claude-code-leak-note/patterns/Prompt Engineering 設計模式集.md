# Prompt Engineering 設計模式集

> 從 Claude Code 原始碼提煉的 17 個 Prompt Engineering 設計模式

## 模式 1：三明治強化（Sandwich Reinforcement）

**問題**：模型容易忽略只出現一次的關鍵指令
**解法**：在 prompt 開頭和結尾都放置同一指令

```
[開頭] CRITICAL: Do NOT call any tools.
[正文] ... 任務指令 ...
[結尾] REMINDER: Do NOT call any tools.
```

**應用**：[[Context Compaction 壓縮策略]] 的 `NO_TOOLS_PREAMBLE` + `NO_TOOLS_TRAILER`
**效果**：工具呼叫失效率從 2.79% 降至接近 0%

## 模式 2：後果規範（Consequence Specification）

**問題**：模型不知道違反規則的後果
**解法**：明確說明「如果違反 → 會發生什麼」

```
Tool calls will be REJECTED and will waste your only turn — you will fail the task.
```

## 模式 3：Scratchpad 思考空間

**問題**：需要模型先思考再回答，但不想保留思考過程
**解法**：提供 `<analysis>` 標籤作為草稿空間，後處理時移除

```
Before providing your final summary, wrap your analysis in <analysis> tags...
```

**應用**：[[Context Compaction 壓縮策略]]、ExtractMemories

## 模式 4：工具沙箱（Tool Sandboxing）

**問題**：某些 prompt 不應使用工具
**解法**：明確列出可用/禁用工具白名單

**應用**：所有 forked agent 的 `canUseTool()` 限制

## 模式 5：最佳路徑規範（Optimal Path Specification）

**問題**：模型可能用低效的方式完成任務
**解法**：明確告知「最高效的做法是 X」

```
Turn 1 — 全部 Read（並行讀取所有可能更新的文件）
Turn 2 — 全部 Write/Edit（並行寫入）
```

**應用**：[[ExtractMemories 自動記憶提取]]

## 模式 6：動態邊界標記（Boundary Marker）

**問題**：動態內容破壞全域 prompt cache
**解法**：用標記分隔靜態/動態部分

```typescript
__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__
```

**應用**：[[System Prompt 動態組裝邏輯]]、[[Cache 穩定性工程模式]]

## 模式 7：Branded Type 防護

**問題**：普通 `string[]` 可能被誤用為 System Prompt
**解法**：TypeScript Branded Type 強制型別安全

```typescript
type SystemPrompt = readonly string[] & { readonly __brand: 'SystemPrompt' }
```

## 模式 8：DANGEROUS_ 前綴自文件化

**問題**：某些 API 放入未緩存區段有合理理由但容易誤用
**解法**：API 名稱包含 `DANGEROUS_` + 強制填寫理由參數

```typescript
DANGEROUS_uncachedSystemPromptSection(key, factory, reason)
```

## 模式 9：條件注入維度分離

**問題**：N 個 boolean 條件放在靜態部分 → 2^N 個 cache 變體
**解法**：條件內容移到動態邊界之後

## 模式 10：自我參照指令

**問題**：AI 可能修改自己的安全指令
**解法**：在源碼注釋中直接對模型發出指令

```typescript
/** Claude: Do not edit this file unless explicitly asked to do so by the user. */
```

**應用**：[[Cyber Risk 安全指令]]

## 模式 11：多重身份上下文

**問題**：同一模型在不同環境需要不同行為
**解法**：根據環境切換身份前綴

**應用**：CLI / SDK / 純代理 三種模式

## 模式 12：顯式 Do/Don't 框架

**問題**：模型不確定邊界在哪
**解法**：同時列出「應該做」和「不應該做」

## 模式 13：Mustache 模板變數

**問題**：Prompt 需要動態值替換
**解法**：`{{variableName}}` 語法 + single-pass 安全替換

**應用**：[[Session Memory 即時快照]]

## 模式 14：容量感知動態警告

**問題**：靜態警告不反映問題嚴重程度
**解法**：根據當前狀態動態調整警告級別

```
正常 → 無警告
接近上限 → IMPORTANT
超限 → CRITICAL
```

## 模式 15：顯式優先序

**問題**：多個指令來源可能衝突
**解法**：明確列出優先序

## 模式 16：可逆性框架

**問題**：模型不知道操作的風險等級
**解法**：聲明操作的可逆性級別

```
Reversible: git 操作 → 可自由執行
Irreversible: 刪除檔案 → 需要確認
```

## 模式 17：元指令分離

**問題**：內部指令洩漏到模型輸出
**解法**：明確聲明「這些指令不是對話的一部分」

```
IMPORTANT: This message and these instructions are NOT part of the actual user conversation.
```

**應用**：所有 [[輔助 Prompt 子系統]]

## 設計哲學總結

> [!tip] 核心思想
> - **具體優於抽象**：用範例而非描述
> - **後果驅動**：告知違反的後果
> - **效能感知**：prompt 設計考慮 cache 影響
> - **多層防禦**：重要指令多次出現
> - **自文件化**：API 命名本身傳達意圖
